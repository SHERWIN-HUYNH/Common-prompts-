Overall this is a well-structured plan — the phase ordering (document-retrieval → rag-chat → chatbot-ui → sync → self-verify) correctly respects the dependency chain you called out yourself (stream error codes must exist in rag-chat before chatbot-ui can reference them), and the "specs own behavior, ADRs own decisions" principle is applied consistently rather than just stated. That said, there are several gaps that will bite you either during self-verify or at implementation time.

## Real gaps worth fixing before Go!

**1. Reranker query contract (1.3) is still underspecified.** You say BGE reranker must receive "the original contextualized/rewritten query string (pre-sparse-fold)" — but you never say whether it gets the *raw* string or the `normalize_for_dense` output (NFC + whitespace collapse). These aren't the same string if whitespace collapse changes anything. Given ADR-0011 introduced two named normalize functions, the spec should bind the reranker input to one of them explicitly (`normalize_for_dense` output, most likely) rather than "original" which is ambiguous between "untouched by any normalization" and "not sparse-folded."

**2. Marker-list heuristic (2.2/2.3) has a coverage gap you haven't flagged as a risk.** `VI_REWRITE_MARKERS` is 8 words. Real traffic will contain many more no-diacritic Vietnamese queries that don't hit this list, meaning the fallback path (`language: unknown`) will still frequently miss VI classification even after this fix — the exact problem ADR-0012 was written to solve. Your "Risk if skipped" table doesn't cover this because it's a risk *even if you don't skip anything* — it's a residual gap in the accepted design. Recommend adding it explicitly to Phase 1.5-style "Known limitations" section for rag-chat, and adding a scenario for telemetry (log `language_resolution_method: heuristic|router|default` so you can measure false-negative rate post-deploy and expand the list later).

**3. Naming collision risk between `VI_REWRITE_MARKERS` (new, for grade/language detection) and the pre-existing `REWRITE_MARKERS` used by `contextualize_query` for noise detection** (from the original workflow doc). If these are meant to be different lists serving different purposes, say so explicitly in the spec text — otherwise whoever implements this will likely just reuse one constant, silently coupling two unrelated concerns (contextualize skip-logic vs. grade language detection) that happen to fire together but shouldn't share failure modes.

**4. Retry-loop ambiguity in the language heuristic.** 2.2 says heuristic runs on `contextualized_query` — but after a rewrite cycle, grading happens against `rewritten_query`/`active_query`, which is the whole point of ADR-0012's query-resolution fix. If the language heuristic only ever looks at `contextualized_query`, and rewrite changes wording (even while preserving language per 2.4), you have two different queries being evaluated by two different mechanisms in the same turn. Spec should state explicitly which query text feeds language resolution on retry N>1 — most consistent choice is probably "whichever query text retrieval used for setting `active_query`", not a static reference.

**5. `_rewrite` / `rewrite_query_node` implementation detail leaking into rag-chat spec (2.4).** Your own guiding principle #1 says specs own *behavior*. Naming specific functions (`ChatbotFlowBuilder._rewrite`, `langgraph_flow.rewrite_query_node`) inside a spec requirement is implementation detail bleeding across the boundary you set for yourself. Move that sentence to `design.md` (you already have a design D4 slot for it) and keep the spec requirement phrased purely behaviorally: "grade-retry rewrite behavior MUST be defined in exactly one place; duplicate/legacy node paths MUST NOT diverge in behavior."

**6. Truncation order in 1.5 is unresolved, not just unspecified.** "Truncate tail when length exceeds `EMBEDDING_MAX_CHARS`" — but truncate *before* or *after* NFC normalization? Vietnamese diacritic characters can be multi-codepoint under NFD before NFC composition; truncating mid-character or before normalization could produce different byte/char counts than truncating after. This interacts directly with the normalization work in 1.1 and deserves its own scenario rather than being a P2 afterthought — recommend bumping it to P1 since it's a correctness question, not just an edge case, once you have real Vietnamese text in the loop.

**7. Golden recall scenario (1.2) doesn't pin the reranker config.** Since `RETRIEVAL_RERANKER` can be `bge` or `none` (different confidence thresholds apply per ADR-0012), the golden recall test's pass/fail could differ depending on which config it runs under. State explicitly which config the golden fixture runs against (probably both, as two scenarios) or your "objective done criteria" claim in the risk table isn't actually objective.

## Minor but worth a pass

- 1.7's health-check exposing `sparse_embedding_model` — worth one line in Phase 5 checklist confirming this isn't a Qdrant payload change (your non-goals list says "no new Qdrant payload field"), just to avoid a false-positive fail during self-verify.
- Phase 5 checklist item "Testability" checks for vague `MAY`/"comparable" language, but doesn't check for 1:1 mapping in the other direction — i.e., no orphaned scenarios that don't trace back to an ADR decision or an identified problem. Add that as a companion check to avoid scope creep sneaking into "hardening."

## Suggested addition to the plan

A "Phase 1.5-equivalent known-limitations subsection" for rag-chat, listing the marker-list coverage gap (#2) and the retry-query ambiguity (#4) as explicit accepted risks with a monitoring hook — mirrors what you already did well for the sparse homophone-collision limitation in Phase 1.5, just extended to Phase 2. This keeps the self-verify "Non-goals respected" check honest: you're not silently under-specifying, you're consciously deferring with a stated reason.

None of these are blockers to starting the edit — they're the kind of gaps that show up as "wait, which string exactly?" mid-implementation. Worth resolving #1, #3, #4, #6 before Go!, since they affect P0/P1 items; #2, #7 can ride along as explicit known-limitations rather than blocking.
